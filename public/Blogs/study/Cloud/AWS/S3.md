# S3 (Simple Storage Service) Core

## 1. Concepts & Architecture

|Target|Key Characteristics|Interview Points|
|-|-|-|
|Object Storage| Stores unstructured data as single objects with unique keys. | Scalable flat structure, not a hierarchical file system.|
|Durability| Guarantees **99.999999999% (11 Nines)** data durability. | Data is automatically replicated across multiple Availability Zones (AZs).|
|Bucket| A globally unique logical container for storing objects. | Name must be unique across all AWS accounts worldwide.|

---

## 2. Storage Classes

- **S3 Standard**
  - **Best for:** Frequently accessed data.
  - **Key Feature:** Low latency and high throughput. Standard choice for active applications.
- **S3 Standard-IA (Infrequent Access)**
  - **Best for:** Data accessed less frequently but requires immediate access when needed.
  - **Key Feature:** Cheaper storage costs than Standard, but incurs a **Retrieval Fee (데이터 인출 비용)** per GB.
- **S3 Glacier Flexible / Deep Archive**
  - **Best for:** Long-term archival data and compliance retention.
  - **Key Feature:** Extremely low storage cost. Retrieval times range from minutes (Flexible) to hours (Deep Archive).

---

## 3. Storage Management & Automation

- **S3 Lifecycle Policies**
  - **Role:** Automates the movement of objects between different storage classes over time to minimize costs.
  - **Scenario:** Moving application logs from Standard $\rightarrow$ Standard-IA after 30 days $\rightarrow$ Glacier after 90 days $\rightarrow$ Auto-deletion after 365 days.
- **S3 Versioning**
  - **Role:** Keeps multiple variants of an object in the same bucket to protect against accidental overwrites or deletions.
  - **Key Feature:** Deleting an object simply applies a **Delete Marker (삭제 마커)**, allowing instant recovery by removing the marker.

---

## 4. Security & Access Control

- **S3 Block Public Access**
  - A centralized bucket-level guardrail that overrides any permissive Object ACLs or Bucket Policies to guarantee zero public internet exposure.
- **Bucket Policy vs. Object ACL**
  - **Bucket Policy:** A JSON-based resource policy attached at the bucket level to manage coarse-grained or cross-account access controls.
  - **Object ACL (Access Control List):** Legacy, fine-grained access control applied to individual files. *AWS now recommends disabling ACLs and using Bucket Policies exclusively.*
- **S3 Presigned URLs**
  - Temporary security tokens granted by an IAM user/role allowing external users to securely upload or download specific objects without requiring AWS credentials.

# Scenario Drill

### Q1. You have terabytes of application logs generated daily. How would you design a cost-effective S3 storage lifecycle architecture for this?
**A:** I would implement **S3 Lifecycle Policies** to shift storage classes based on access frequency. Logs will be initially stored in **S3 Standard** for immediate troubleshooting access. After 30 days, the policy will transition them to **S3 Standard-IA** to reduce storage costs. After 90 days, they will move to **S3 Glacier Deep Archive** for cheap long-term compliance retention, and finally execute an automated permanent deletion rule after 1 year.

---

### Q2. A critical object in an S3 bucket was accidentally deleted by a team member. How do you recover this file, and how do you prevent this risk in the future?
**A:** If **S3 Versioning** was enabled, the deletion simply created a **Delete Marker** on top of the object stack. I can recover the file instantly by turning on "Show versions" in the console and deleting that specific Delete Marker. To prevent accidental deletions moving forward, I would enable **MFA Delete (다중 인증 삭제)** on the bucket, which requires multi-factor authentication credentials to permanently purge any object version.

---

### Q3. Your frontend application needs to let users upload profile images directly to an S3 bucket securely, without exposing any long-lived AWS access keys in the browser code. How would you implement this?
**A:** I would utilize **S3 Presigned URLs**. The frontend client will first request an upload token from our backend API server. The backend, backed by an authorized IAM Role, will programmatically generate a time-restricted Presigned URL for a specific S3 key path using the AWS SDK and return it to the client. The frontend can then execute a direct HTTP `PUT` request to that URL to upload the image securely.
